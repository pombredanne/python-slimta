slimta.smtp.auth¶

Package implementing the ability for client and server SMTP sessions to authenticate.

For servers, use Auth as a base class and override the verify_secret() and get_secret() methods as needed. To modify which auth mechanisms are advertised in clear- and encrypted channels, you can override the get_available_mechanisms() method as well. Your Auth sub-class may be passed in to your Server or SmtpEdge.

exception slimta.smtp.auth.CredentialsInvalidError¶

Bases: slimta.smtp.auth.ServerAuthError

Thrown when a clients credentials were not correct.

class slimta.smtp.auth.Auth(session)¶

Bases: object

Base class that handles the authentication for an SMTP server session. This class should be inherited with its methods overridden as necessary.

Parameters:session (Server) – An active server session object.
verify_secret(authcid, secret, authzid=None)¶

For SMTP server authentication, this method should be overriden to verify the given secret for mechanisms that have access to it. If secret is invalid for the given identities, CredentialsInvalidError should be thrown.

Parameters:
  • authcid – The authentication identity, usually the username.
  • secret – The secret (i.e. password) string to verify for the given authentication and authorization identities.
  • authzid – The authorization identity, if applicable.
Returns:

A representation of the identity that was successfully authenticated. This may be authcid, authzid, a tuple of both, or any alternative.
Raises :

CredentialsInvalidError
get_secret(authcid, authzid=None)¶

For SMTP server authentication mechanisms such as CRAM-MD5, the client provides a hash of their secret credentials, and thus the server must also have access to the secret credentials in plain-text. This method should retrieve and return the secret string assocated with the given identities. This method should raise CredentialsInvalidError only if the given identities did not exist.

Parameters:
  • authcid – The authentication identity, usually the username.
  • authzid – The authorization identity, if applicable.
Returns:

A tuple of the retrieved secret string followed by a representation of the identity that was authenticated. This may be authcid, authzid, a tuple of both, or any desired alternative.
Raises :

CredentialsInvalidError
get_available_mechanisms(connection_secure=False)¶

Returns a list of mechanism classes that inherit ServerMechanism that are available on the session. This usually depends on whether the connection is secure, as plain-text mechanisms should not be used unencrypted.

Unless overridden, this method will return attempt to use all built-in auth mechanisms.

Parameters:connection_secure (boolean) – True if the current connection has been secured with TLS, False otherwise. This flag can be used to adjust which mechanisms are available to unencrypted sessions.
Returns:List of available mechanism classes.
class slimta.smtp.auth.ServerMechanism(verify_secret, get_secret)¶

Bases: object

Base class of server-side SASL authentication mechanism implementations.

Parameters:
  • verify_secret – Function used by an SMTP server to verify credentials given by the client.
  • get_secret – Function used by an SMTP server to retrieve the secret (password) string for comparison with credentials given by the client.
secure = False¶

This static flag should be overidden by sub-classes to specify whether the implementation is safe to use over unencrypted channels. By default, unsafe mechanisms will not be advertised until TLS is negotiated.

name = None¶

This static string should be overriden by sub-classes to specify the SASL name that identifies this mechanism. This string will be used in the SMTP session. Custom mechanisms should be prefixed with X.

requires_get_secret = False¶

This static flag should be overridden by sub-classes if use of the mechanism requires direct access to the original password, such as CramMd5.

If an Auth sub-class does not override get_secret(), mechanisms that have set this flag to True will be automatically removed.

send_challenge_get_response(io, challenge_str)¶

This method can be used in server_attempt() implementations to send intermediate SASL challenge strings to the client. The challenge string will be automatically prefixed with the 334 code to indicate it expects a response from the client. The cancellation response * is also handled by this method, which raises an internal exception to break out of the server_attempt() method.

Parameters:
  • io (IO) – The underlying IO object.
  • challenge_str – The challenge string to send to the client.
Returns:

The string sent back by the client in response to the challenge.
server_attempt(io, initial_response)¶

Communicates back-and-forth with the connected client to negotiate authentication, based on the client’s auth string. This method must be overidden by sub-classes to be used by server-side SMTP sessions.

Parameters:
  • io (IO) – The underlying IO object.
  • initial_response – The initial string sent by the client along with its AUTH command.
Returns:

A representation of the identity that was successfully authenticated. This may be authcid, authzid, a tuple of both, or any alternative.
Raises :

CredentialsInvalidError
class slimta.smtp.auth.ClientMechanism¶

Bases: object

Base class of client-side SASL authentication mechanism implementations. Sub-classes will not be instantiated and should only use static members and class methods.

name = None¶

This static string should be overriden by sub-classes to specify the SASL name that identifies this mechanism. This string will be used in the SMTP session. Custom mechanisms should be prefixed with X.

classmethod send_response_get_challenge(io, response_str='', first=False)¶

This method can be used in client_attempt() implementations to send response strings to the server and wait for its challenge. It is up to the implementation to check if this challenge is 334 (and thus requires another response) or if it is the final reply.

Parameters:
  • io (IO) – The underlying IO object.
  • response_str – The response string to send to the server.
  • first (bool) – Every SASL implementation must first send a response with this argument as True to initiate the authentication request. Subsequent calls should use False.
Returns:

The challenge or final Reply sent back by the server after the response.
classmethod client_attempt(io, authcid, secret, authzid)¶

Communicates back-and-forth with a server to negotiate authentication, based on the desired credentials. This class method must be overidden by sub-classes to be used by client-side SMTP sessions.

Parameters:
  • io (IO) – The underlying IO object.
  • authcid – The authentication identity, usually the username.
  • secret – The secret (i.e. password) string to send for the given authentication and authorization identities.
  • authzid – The authorization identity, if applicable.
Returns:

Reply object received by the final authentication negotiation response from the server.
